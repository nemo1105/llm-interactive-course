# 课程目录

## 目录标准

本目录记录已接受的课程顺序，用于从推理接口向外解释 LLM 和智能体（agent）基础。它是后续章节文档、动画规格和页面开发的事实来源。

课程目标是让学习者由浅入深理解 LLM 应用和 Agent 系统：先理解一次推理接口，再理解消息、工具、token、参数、输出、缓存、提示词和上下文工程；随后用最小 Agent Loop 建立核心准则，再展开 MCP、RAG、memory、skills、错误处理、运行环境、文件、上下文压缩、计划任务、通道、连接器、权限、人机协作、Agent 协作和评估闭环。

本目录批准章节顺序、章节标题、原理讲法方向和演示主题进入页面制作。真实外部 API、真实账号授权、真实邮件或生产动作不在页面实现范围内，均使用 mock payload 展示工程机制。

## 章节

### 01. [最小的一次 LLM 对话](chapters/01-minimal-llm-conversation.md)

讲什么：从一条用户消息开始，拆开应用构造请求、模型生成输出、应用展示结果的最小链路。

如何讲原理：所有后续技术都不是独立魔法，而是在改变、扩展、约束或评估同一次推理接口调用。

实例：普通改写请求和上海客户现场出行天气问题，分别展示直接文本响应和 mock get_weather 工具调用。

完整知识点：

- 用户输入先进入应用层。
- 应用构造模型请求。
- 模型输出文本或工具调用请求。
- 工具由应用执行，结果再写回模型上下文。

### 02. [多轮对话的真实机制](chapters/02-conversation-history-context.md)

讲什么：在第一章普通对话链路上增加一轮追问，展示多轮对话的关键不是模型记住网页聊天，而是应用服务器把历史消息放进下一次模型请求。

如何讲原理：模型每次只能基于本次调用可见上下文回答；多轮对话的“记忆感”来自应用服务器在第二次请求中显式带回上一轮 user 和 assistant 消息。

实例：第一轮用户问“明天下午我要去上海客户现场，出门要带伞吗？”，助手回答上海天气和带伞建议；第二轮用户追问“那我还需要提前多久出门？”，应用服务器把第一轮问答和第二轮追问一起发给模型。

完整知识点：

- 时序图参与者仍然是用户、应用服务器和大模型。
- 历史上下文不是参与者，而是第二次模型请求 payload 里的 messages。
- 如果第二次请求只发送当前追问，模型看不到“那”指向上一轮行程。
- 显式历史重传会增加输入 token，并受上下文窗口限制。

### 03. [消息、Role 与指令层级](chapters/03-messages-roles-instruction-hierarchy.md)

讲什么：把聊天记录展开成 system、developer、user、assistant、tool 消息结构，说明指令层级如何影响行为。

如何讲原理：模型看到的是带 role 和边界的消息序列；高层指令、用户输入和工具结果不是同一种信息。

实例：用户要求忽略规则并泄露天气工具密钥，developer 指令限制只能使用公开天气字段，模型拒绝泄露并继续安全范围内的查询。

完整知识点：

- system 和 developer 指令高于 user 请求。
- assistant 消息可包含文本或工具调用。
- tool 消息来自应用执行结果。
- 指令和数据必须隔离，避免把用户资料当规则。

### 04. [工具调用的本质](chapters/04-tool-calling-essentials.md)

讲什么：讲清楚工具调用解决什么问题，以及模型、应用、工具运行时各自负责什么。

如何讲原理：模型输出的是结构化工具调用请求，应用才是真正执行工具的一方。

实例：上海张江下午三点到五点是否适合步行去客户现场，模型通过 get_weather 参数请求天气，应用执行工具并回写结果。

完整知识点：

- 工具声明包含 name、description 和 parameters schema。
- 工具描述影响模型是否选择工具。
- tool_choice 可以自动、强制或禁止工具。
- 工具结果必须回写后，模型才能基于结果生成回答。

### 05. [用 get_weather 拆解完整工具调用链路](chapters/05-get-weather-tool-chain.md)

讲什么：用一个 mock 天气工具覆盖正常路径、缺参、工具报错和异常数据。

如何讲原理：工具调用链路需要同时设计成功路径和失败路径，否则模型会在不可靠数据上继续生成。

实例：上海张江客户现场出行建议中，正常返回降水概率；城市缺失时追问；工具不可用时说明限制；降水概率 180% 时工程层拦截。

完整知识点：

- 参数缺失不应由模型凭空补全关键事实。
- 业务错误可以回写模型继续追问。
- 系统错误和异常数据应由工程层分类处理。
- 最终回答必须说明依据和不确定性。

### 06. [Token、上下文窗口、成本与延迟](chapters/06-tokens-context-cost-latency.md)

讲什么：解释模型输入输出的计量单位，以及长上下文为什么会影响成本、速度和体验。

如何讲原理：上下文中的每一块内容都会消耗 token 预算，输出和推理预算也会影响延迟。

实例：上海张江多轮对话逐步加入历史、工具 schema 和天气结果，预算条显示输入、输出、reasoning token 的变化。

完整知识点：

- Token 是模型处理文本和多模态内容的基本计量单位。
- 上下文窗口限制一次调用可见信息量。
- 输入越长，成本和延迟通常越高。
- 流式输出改善首字体验，但总生成仍需要时间。

### 07. [推理接口控制参数](chapters/07-inference-control-parameters.md)

讲什么：解释一次推理调用中的常见参数分别控制什么，以及它们和提示词的边界。

如何讲原理：参数是推理接口层面的行为控制，不是提示词、上下文和工具的替代品。

实例：把客户拜访三条记录整理成销售经理摘要，通过 temperature、seed、verbosity、reasoning effort 和 max output tokens 对比输出变化。

完整知识点：

- temperature 和 top_p 影响采样随机性。
- seed 帮助复现实验，但仍要记录模型和参数版本。
- verbosity 和 max output tokens 影响输出详细度与截断。
- parallel tool calls 和 truncation 影响工具场景与上下文超限处理。

### 08. [推理输出形态](chapters/08-inference-output-shapes.md)

讲什么：说明模型输出不只有最终自然语言，还包括结构化输出、流式片段、推理摘要和工具调用。

如何讲原理：不同输出形态适合不同产品界面，应用要按 output item 类型消费模型输出。

实例：客户会议记录既可以生成一段总结，也可以生成 summary、risks、next_actions、owner、due_date 字段并渲染成任务列表。

完整知识点：

- 文本输出适合直接展示。
- 结构化输出适合应用渲染和后续处理。
- JSON schema 限制字段类型和枚举。
- 流式输出和工具调用输出有不同生命周期。

### 09. [前缀缓存：多轮对话和长上下文的必备优化](chapters/09-prefix-caching.md)

讲什么：讲清楚 prompt 或 prefix caching 如何降低稳定上下文的重复处理成本和延迟。

如何讲原理：稳定内容放前面、动态内容放后面，可以提高前缀缓存命中；缓存不改变模型看到的语义。

实例：A 客户续约风险助手中，固定 developer 规则、B公司 2026年SLA合同摘要和 CRM 工具列表放在前部，用户追问放后部。

完整知识点：

- 缓存命中依赖请求前缀稳定。
- 大工具列表和稳定合同摘要适合缓存。
- 动态历史插到前面会破坏命中。
- prompt_cache_key 只帮助路由相似请求，不改变语义。

### 10. [提示词工程（Prompt Engineering）](chapters/10-prompt-engineering.md)

讲什么：从 role、前缀缓存和上下文出发，解释提示词如何明确目标、约束、格式和示例。

如何讲原理：提示词改变的是模型当前调用中可见的任务说明和示例，不能弥补缺失资料或替代工程校验。

实例：把销售同事的客户拜访记录改成销售经理 30 秒内能读懂的行动摘要，逐层加入目标、约束、格式、few-shot 和反例。

完整知识点：

- 目标让模型知道产物用途。
- 约束限制事实来源和边界。
- 格式让应用和用户更容易消费。
- few-shot 和反例提升稳定性，但不能保证事实正确。

### 11. [上下文工程（Context Engineering）](chapters/11-context-engineering.md)

讲什么：把 prompt 扩展为整个上下文构造工程，关注放什么、放多少、顺序、隔离、压缩和来源。

如何讲原理：模型的行为由完整上下文塑造；上下文工程决定模型能看到什么、如何区分规则和资料、如何引用依据。

实例：A 客户问 B公司 2026年SLA合同赔付条款是否适用于延迟交付，正确组织合同片段、交付记录、客户邮件和开发者规则。

完整知识点：

- 指令和不可信资料要隔离。
- 相关资料要带来源和可信度。
- 过多上下文会增加成本并干扰重点。
- 稳定上下文顺序也影响前缀缓存。

### 12. [从工具调用到最小 Agent Loop](chapters/12-minimal-agent-loop.md)

讲什么：把 Agent 还原为模型调用、工具执行、观察写回和下一轮调用的循环，并融入“下一步如何决定”的核心原理。

如何讲原理：Agent 并不神秘。每一步都是模型基于当前上下文输出下一步，工程层负责执行、约束、拦截和停止。

实例：规划明天下午去上海张江客户现场的出发时间，模型先请求查天气，再请求查日历；当用户要求直接发外部邮件时，工程层按权限规则拦截。

完整知识点：

- 模型决定下一步的依据来自 system/developer 提示词、用户目标、工具列表、历史观察、环境和权限。
- 工具结果写回后会改变下一轮模型调用的上下文。
- 最大轮数、工具白名单和审批 gate 让循环可控。
- 后续 RAG、memory、skills、错误处理、环境和协作都必须遵循这个准则。

### 13. [MCP：工具和上下文的标准化连接](chapters/13-mcp-tools-context-standardization.md)

讲什么：在 RAG 前讲 MCP，说明它如何标准化工具、资源和 prompt 的连接，解决每个应用重复封装外部能力的问题。

如何讲原理：MCP 是 host/client/server 之间的协议层，模型仍然通过 host 提供的工具和上下文使用外部能力。

实例：日历 MCP server 暴露 list_events 工具、“今天日程”resource 和会议纪要 prompt，host 发现后提供给模型使用。

完整知识点：

- MCP tools 暴露可执行动作。
- MCP resources 暴露可读取资料。
- MCP prompts 暴露可复用提示模板。
- 权限、审批和质量仍由 host 与工程层负责。

### 14. [RAG 基础：把外部知识放进推理接口](chapters/14-rag-basics.md)

讲什么：讲两种基础 RAG 使用方式：工程侧先检索后插入，以及模型通过检索工具自行生成检索词/句。

如何讲原理：RAG 是把外部资料在调用时提供给模型，不是让模型永久学会文档；检索可以由工程层发起，也可以作为工具由模型发起。

实例：A 客户问 B公司 2026年SLA合同中服务可用性低于 99.5% 是否有赔付条款，分别演示外部插入和 search_contract_chunks 工具检索。

完整知识点：

- 离线阶段包含切分、embedding 和索引。
- 在线阶段可以由应用先召回资料再拼装进请求。
- 也可以提供检索工具，让模型生成检索词或句子并调用向量相似度检索。
- 最终回答需要引用来源，并区分检索失败和生成失败。

### 15. [RAG 优化](chapters/15-rag-optimization.md)

讲什么：讲真实系统中如何提高召回质量、排序质量、拼装质量和引用可信度。

如何讲原理：RAG 优化先要判断失败发生在召回、过滤、重排、拼装还是生成阶段。

实例：A 客户问计划维护窗口是否排除 SLA 赔付，基础 RAG 错召回到 2025 合同，通过元数据过滤、查询改写、多查询、重排和引用校验修正。

完整知识点：

- 元数据过滤避免召回错误客户、年份或权限资料。
- 查询改写和多查询提升召回覆盖。
- 重排把核心条款放到上下文前部。
- 父子块和引用校验减少断章取义。

### 16. [记忆（Memory）：跨会话保存和读取上下文](chapters/16-memory-cross-session-context.md)

讲什么：把 memory 去神秘化，明确它是应用维护的跨会话上下文系统，并讲两种实现方式。

如何讲原理：长期记忆效果来自工程侧存取和上下文注入，或来自模型按系统提示词使用记忆工具检索与更新。

实例：用户上周说以后写项目汇报先列风险再列进度。新会话整理 LLM 分享项目进展时，演示工程外部注入和 search_memory/upsert_memory 工具两条路径。

完整知识点：

- 实现一：像传统 RAG 一样，工程层先检索相关 memory 并插入上下文。
- 实现二：系统或开发者提示词明确告知模型必须使用记忆工具实现长期记忆效果。
- 记忆工具可包含检索、写入、更新和删除。
- Memory 不是模型天然记住，也不是无条件保存全部聊天。

### 17. [技能（Skills）：渐进式暴露的工作方法包](chapters/17-skills-progressive-disclosure.md)

讲什么：把 Skills 作为核心章节，讲清楚技能是文档、脚本、二进制、模板等文件组合，并通过渐进式暴露进入模型上下文。

如何讲原理：Skill 不是模型天然能力，而是系统/开发者提示词告诉模型有哪些技能和入口路径，模型按需读取文件，把文件内容变成上下文后再工作。

实例：用户一句“用会议纪要技能处理这份材料”，模型读取技能入口文件、按目录描述继续读取流程文档和检查脚本；另一个例子组合文档技能和表格技能完成客户会议纪要与行动项同步。

完整知识点：

- 系统/开发者提示词明确技能是什么、如何使用、有哪些技能、入口文件绝对路径在哪里。
- 技能加载通过文件读取完成，模型阅读技能文件后才获得上下文。
- 渐进式加载是入口文件指向下一层文档、脚本或模板，模型按任务读取需要的部分。
- 技能减少用户输入，把稳定流程从聊天里迁移到可复用文件。
- Memory 机制也可以用类似方式：提示词说明规则，配套工具或 CLI 做存储检索。

### 18. [智能体工程化：错误如何处理](chapters/18-agent-error-handling.md)

讲什么：区分工程层错误、业务结果、权限结果和临时工具失败，决定哪些要拦截，哪些回写模型。

如何讲原理：不是所有错误都应该让模型继续处理；工程层负责保护系统边界，模型负责基于可见业务结果继续推理或追问。

实例：查询 CRM 中 A 客户本季度续约风险，token 失效需要重新授权，客户不存在可回写模型追问，API 连续超时则停止循环。

完整知识点：

- 认证、SDK、模型 API 连续失败属于工程层处理。
- 业务无结果和参数不明确可以结构化回写模型。
- 权限不足但可申请授权可以让模型解释授权路径。
- 错误分类会影响重试、替代工具、停止和用户说明。

### 19. [智能体工程化：运行环境如何进入上下文](chapters/19-agent-runtime-environment-context.md)

讲什么：讲环境信息既可以固定写入 developer prompt，也可以通过环境查询工具进入上下文，并引出多环境工具执行。

如何讲原理：模型不知道自己在哪个环境里工作，除非环境信息被写入上下文或通过工具查询；多环境应由工程层维护当前环境，而不是让每个工具都带环境 id。

实例：本地环境上传文件到云端环境，切换到云端后用强硬件处理 AI 图片或视频生成，所有工具按当前环境路由执行。

完整知识点：

- 环境信息可以固定在 developer prompt 中提供。
- 环境信息也可以通过 get_environment、list_files、read_policy 等工具查询。
- 多个环境要通过 switch_environment 工具切换当前环境。
- 业务工具无需都写 environment_id 参数，工程层按当前环境执行。

### 20. [智能体工程化：文件、目录与用户上传内容](chapters/20-agent-files-and-uploads.md)

讲什么：重点介绍图片类可直接进入模型上下文的文件，以及模型可见元数据后通过工具脚本操作的文件。

如何讲原理：上传文件不等于模型读完文件；文件要么被工程能力转换进上下文，要么作为可操作对象暴露给模型再由工具读取。

实例：图片用 base64 和 URL 两种方式进入上下文；PDF、Word、PPT、Excel、JS/Go 源码以文件清单、路径、大小、类型、摘要和工具读取方式进入模型可见范围。

完整知识点：

- 图片可以由工程层转成 base64 或 URL 作为多模态上下文。
- 办公文件和代码文件通常先作为元数据进入上下文，再由工具解析、检索或局部读取。
- 模型需要知道文件位置、名称、大小、类型和可用工具。
- 回答必须区分已读取内容和未检查内容。

### 21. [智能体工程化：上下文超过上限怎么办](chapters/21-agent-context-overflow.md)

讲什么：讲模型上下文从 32K 到 1M 仍会遇到上限，核心处理方式是压缩、摘要、保留来源和按需回查。

如何讲原理：压缩必然丢信息，丢什么不能靠固定规则通吃，应让模型结合当前任务和开发者规则生成压缩后的历史。

实例：从 12 份 A 客户访谈、B公司 2026年SLA合同和 CRM 摘要整理续约风险，模型生成带来源的压缩历史，再结合固定 system/developer prompt 和新用户消息继续推理。

完整知识点：

- 上下文窗口再大也会被长任务、文件和工具结果耗尽。
- 自动截断可能丢掉关键约束。
- 模型可根据任务目标总结压缩历史，但要保留来源和关键否定约束。
- 压缩后上下文与固定提示词、新用户消息共同组成下一次调用。

### 22. [智能体工程化：计划任务与定时执行](chapters/22-agent-scheduled-tasks.md)

讲什么：讲计划任务不是 agent 自己写 todo，而是模型通过工具创建定时任务，工程层到时间发起新的模型调用。

如何讲原理：计划任务由工程系统持久化和触发；触发时，模型编写的任务内容作为用户消息，结合系统提示词进入一次新的推理调用。

实例：用户要求每天早上 10 点检查邮箱并更新当天日程，模型调用计划任务工具创建任务；到点后工程层触发模型调用，模型再调用邮箱和日历工具完成更新。

完整知识点：

- 工程层提供计划任务工具或 CLI。
- 计划任务列表由系统维护，不靠聊天历史。
- 执行时依旧是一次模型调用。
- 任务内容由模型编写并作为触发时的用户消息。
- 执行过程中仍可调用邮箱、日历、通道和审批工具。

### 23. [通道（Channels）：消息从哪里来、往哪里去](chapters/23-channels-message-surfaces.md)

讲什么：讲飞书、钉钉、WhatsApp、自定义 API 等通道如何透传用户消息和模型响应，并提供通道专属开发者提示词。

如何讲原理：通道是消息承载面，不是模型能力本身；不同通道需要告诉模型如何发送图片、文件、表单和富文本。

实例：同一个 A 客户风险提醒，通过飞书发送交互表单，通过 WhatsApp 发送短消息，通过自定义 API 返回结构化 JSON。

完整知识点：

- 通道负责接收用户消息并投递模型响应。
- 每个通道可以有自定义 developer prompt。
- 通道提示词说明图片、文件、按钮、表单等渲染能力。
- 模型输出仍由工程层适配成通道消息。

### 24. [连接器（Connectors）：授权外部系统能力](chapters/24-connectors-authorized-tools.md)

讲什么：讲邮箱、日历、社交软件等第三方平台需要凭证，连接器通过 OAuth 或其他认证方式让用户预授权。

如何讲原理：连接器把用户授权、凭证管理和平台 CLI/工具封装起来，模型通过工具发现授权状态并请求授权或使用平台能力。

实例：模型要操作用户邮箱和日历，发现邮箱未授权后调用 request_connector_auth，让用户完成 OAuth，再通过自研 CLI 工具读取邮件和更新日历。

完整知识点：

- 三方平台如果提供 OAuth 或认证方式，可以封装为连接器。
- 用户可以预授权，也可以在模型发现未授权时再授权。
- 连接器工具要暴露授权状态、授权请求和平台功能。
- 凭证不进入模型上下文，模型只看授权状态和可用动作。

### 25. [权限、审批与高风险动作](chapters/25-agent-permissions-approval.md)

讲什么：讲风险操作的拦截必须由工程层实现，尤其是外部发送、删除、支付和生产变更。

如何讲原理：模型可以请求高风险工具，但工程层必须在工具执行前识别风险、暂停并展示确认弹窗，确认后才执行。

实例：整理客户欠款邮件并拟催款邮件，模型可以写草稿；真正发送前，工程层展示收件人、主题、正文和附件，请用户确认。

完整知识点：

- 只读、写草稿、外部发送、删除、支付、生产操作风险等级不同。
- 高风险工具在执行前被工程层拦截。
- 确认弹窗展示动作、对象、影响、是否可回退。
- 用户确认后才调用真实工具。

### 26. [人和 Agent 的协作](chapters/26-human-agent-collaboration.md)

讲什么：用“环”解释 AI 在真实工作中的介入方式：AI 做多数弧线，人只在少数关键点决策。

如何讲原理：AI 转型成功的典型场景，是一件事的大部分环节由 AI 完成，人保留目标、授权、判断和最终决策点。

实例：复审 A 客户续约风险报告：AI 读取资料、核对合同、整理建议，人只在风险升级、对外沟通和最终口径上做决策。

完整知识点：

- 没有 AI 时整个环由人完成。
- 有 AI 后，AI 在特定环节介入并完成工作。
- 当 AI 做完它能做的所有环节，人只做关键决策点，就是典型转型成功案例。
- 界面要让人看到 AI 已做工作、证据和待决策点。

### 27. [Agent 之间的协作](chapters/27-multi-agent-collaboration.md)

讲什么：在人和 Agent 协作环基础上进一步讲多个 Agent 如何通过工具调用、通讯工具、Agent 发现工具和流程描述形成组织中的“流”。

如何讲原理：Agent 之间通讯的核心仍是工具调用。工程层提供通讯工具、Agent 发现工具和工作流程描述，让 Agent 知道该找谁、怎么交互、什么时候请人决策。

实例：运营 Agent 发现需求后通过通讯工具找产品 Agent 多轮澄清；需要人决策时各自通过通道通知运营和产品经理；确认后产品 Agent 再把需求转给开发 Agent 评审。

完整知识点：

- 多个“人和 Agent 的协作环”可以被 Agent 通讯串成流。
- 被交互的 Agent 可以自行处理，也可以通过通道找人决策。
- 工程层要提供 Agent 发现、通讯工具和工作流程描述。
- 学员要知道自己的工作是开发这些机制，让 Agent 能工作、更好地工作。

### 28. [观测、安全、评估与进化](chapters/28-observability-safety-evaluation-evolution.md)

讲什么：把前面所有机制收束到可观测、可评估、可持续改进的工程闭环中。

如何讲原理：上线后的 LLM 和 Agent 系统需要 trace、失败分类、安全拦截、eval set、回归测试和反馈闭环。

实例：上海客户现场出行 Agent 展示成功 trace、天气工具异常、外部资料 prompt injection、越权工具调用被工程层拦截和 eval 回归。

完整知识点：

- trace 记录用户输入、模型请求、工具调用、结果、token、latency 和 cost。
- prompt injection 需要上下文隔离和权限策略共同处理。
- 越权工具调用必须由工程层拦截。
- eval set 覆盖成功、失败、缺参、工具异常、权限不足和长上下文截断。
