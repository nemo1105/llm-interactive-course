# 01. 最小的一次 LLM 对话

## 复审边界

本文档是已接受的章节产品内容合同。它批准第一章页面使用本文档中的学习目标、概念边界、交互方向、具体对话示例和可视化调用链。

页面实现可以使用具体城市、天气数据和真实感对话示例，但仍不得接入真实外部 API 或暗示 mock 数据来自实时天气服务。

第一章采用固定章节路由：

- `/chapters/01` 是章节首页，展示章节名称和本章要讲什么。
- `/chapters/01/demos/direct` 是普通对话演示页。
- `/chapters/01/demos/streaming` 是流式输出演示页，复用普通对话中的改写请求。
- `/chapters/01/demos/tool-call` 是包含 `get_weather` 的工具调用演示页。
- `/chapters/02` 可以作为下一章预留路由，但内容必须保持中性占位。

## 学习目标

学习者完成本章后，应能对一次 LLM 推理调用建立具体心智模型：

- 用户在应用中输入一条消息，应用先把它记录为带会话 id 的应用内部消息。
- 应用把这条输入转换成模型请求。
- 模型产生输出。
- 应用展示直接文本回答、边读流式片段边更新助手气泡，或先走完工具调用路径再展示最终回答。

本章要建立一个基础认识：后续概念不是彼此分离的新魔法，而是在改变、支持、约束、扩展、评估或工程化同一个推理接口。

## 前置知识

- 学习者只需要理解普通聊天交互。
- 本章不要求听众预先理解 token、role 层级、工具 schema、MCP、RAG、记忆（memory）、评估或智能体循环（agent loop）。
- 后续概念只有在解释调用路径重要性时才可作为预告出现，不应要求听众提前理解其机制。
- 本章页面示例使用已确认的改写请求和上海出行天气问题。

## 核心概念

- 用户输入：学习者输入或提交的消息。
- 应用层：接收输入、构造请求、发送给模型并处理模型输出的软件层。
- 应用消息：用户消息进入应用后的最小内部记录，只表达会话归属和原始用户消息内容，不等同于模型 API 请求。
- 模型请求：应用发送给模型的结构化调用。
- 模型输出：模型返回的结果，可以是自然语言文本，也可以是工具调用请求。
- 直接文本响应：最简单的路径，模型输出可以直接展示给用户。
- 流式输出：模型响应以多个增量片段返回，应用在循环读取和处理片段时逐步更新同一个助手气泡。
- 工具调用请求：模型输出的一种形态，表示它希望应用用指定参数执行某个命名工具。
- mock 工具执行：用于教学的本地模拟函数调用，不依赖真实外部 API。
- 工具结果回写：应用把工具结果放回模型上下文，让模型继续生成最终面向用户的回答。
- 开发者 trace：用于检查聊天界面背后应用行为的可视化记录。

## 教学流程

1. 从一个熟悉的聊天瞬间开始：学习者输入“我这段话有点绕，帮我改得更清楚……”，然后看到助手改写。
2. 揭示聊天框不是完整系统；应用需要先构造模型请求。
3. 展示直接文本路径：
   - 用户输入进入应用，成为带会话归属的应用内部消息；
   - 应用构造请求；
   - 模型返回自然语言文本；
   - 应用展示文本。
4. 展示同一改写请求的流式输出路径：
   - 用户输入进入应用，成为带会话归属的应用内部消息；
   - 应用请求模型时打开流式返回；
   - 应用循环读取模型返回的文本片段；
   - 每个片段被追加到同一个助手气泡，直到最终完成。
5. 展示使用 mock `get_weather` 的工具调用路径：
   - 用户提出“明天下午我要去上海客户现场，出门要带伞吗？”；
   - 应用先接收并记录本轮应用内部消息；
   - 应用在请求中提供可用的 mock 天气工具；
   - 模型返回工具调用请求，而不是最终天气回答；
   - 应用执行 mock 工具；
   - 应用把工具结果写回模型；
   - 模型生成最终自然语言回答；
   - 应用展示回答。
6. 对比三条路径并点出稳定原则：它们仍然围绕同一个推理调用组织；流式输出改变的是应用接收和展示模型文本的节奏，工具使用则会在模型输出之间增加一个由应用控制的执行步骤。

## 交互与演示想法

- 章节首页：顶部展示第一章名称，正文说明本章要讲什么，并提供进入演示的入口。
- 演示页布局：遵循 `AGENTS.md` 的全局演示结构。
- 普通对话演示页：展示真实改写请求和助手改写结果。
- 流式输出演示页：复用普通对话的真实改写请求，展示助手气泡随模型流式片段逐步更新。
- 工具调用演示页：展示上海出行问题、`get_weather` 工具调用、mock 天气返回和最终带伞建议。
- 时序图：把用户、应用、模型、工具作为参与方，按事件展示消息传输方向。
- 普通对话演示必须分步展示用户提交、应用请求模型、模型响应、应用展示四个步骤。
- 流式输出演示必须分步展示用户提交、应用请求模型、应用循环读取和处理流式片段、应用完成展示；左侧助手气泡必须在多步中逐步累积文本。
- 流式输出演示右侧必须使用循环标记包住重复的流式读取及处理步骤，帮助学习者看见同一段逻辑会对多个增量片段反复执行。
- 工具调用演示必须分步展示用户提交、应用声明工具、模型请求工具、应用执行工具、工具返回、应用回写、模型最终响应、应用展示八个步骤。
- 工具调用演示中，左侧可以停留在通用“思考中”状态，右侧继续逐步推进多次传输。
- 本章右侧事件展示应用内部消息、完整模型请求、模型响应、工具参数、工具返回结果和二次模型请求；首条用户到应用的 `发送消息` 不是官方模型 API，请求大模型的步骤才展示 Chat Completions 与 Responses API 两种格式。
- 流式输出演示的模型请求 payload 必须在 Chat Completions 与 Responses API 两种格式中都显式包含 `stream: true`；流式片段 payload 应区分 Chat Completions 的 `chat.completion.chunk` 与 Responses API 的 typed SSE event。
- 本章 Responses API 样例采用无状态上下文重放方式；工具结果回写时显式带回完成最终回答所需的用户输入、模型工具调用和工具结果上下文，不依赖平台托管的上一轮响应指针。
- 导航：章节页右上角提供下一章、进入演示、返回章节首页和演示间切换入口。

## 边界与常见误解

- 本章不讲 token 计量；第 06 章负责 token、成本和延迟细节。
- 本章不讲 role 层级；第 05 章负责 role 和指令优先级。
- 本章不讲完整工具 schema 设计；第 03 章负责通用工具机制，第 04 章负责完整 `get_weather` 拆解。
- 本章不讲 MCP；第 12 章负责工具和上下文的协议标准化。
- 本章不讲 RAG 或 memory；后续章节负责外部知识和跨会话上下文。
- 不要暗示模型自己执行工具。模型请求工具调用，应用执行工具。
- 不要暗示工具结果会变成模型永久学会的知识。它只是提供给下一次模型调用的上下文。
- 不要暗示每个用户问题都需要工具。直接文本路径仍然是最简单的有效路径。

## 可复用组件需求

- 用于用户消息和 assistant 消息的聊天界面。
- 用于章节首页的介绍区域和演示入口卡片。
- 用于本章章节右上角导航的按钮组。
- 用于展示本章参与方、消息方向和事件描述的时序图。
- 用于展示本章完整 payload 的传输数据浮层。
- 用于存放章节首页、演示页、聊天帧、时序参与者、时序消息、step 序列和 payload 的 `DemoSpec` 结构化内容数据。
- 第一章演示页应复用通用 `DemoPlayer`，不得保留第一章专用时序图运行时。

## 内容复审验收标准

- 本章解释一条完整的直接文本响应路径。
- 本章解释一条完整的流式输出路径，且复用已确认的普通改写示例。
- 本章解释一条完整的 mock 工具调用路径，并使用具体城市与天气数据。
- 本章首页和演示页路由为 `/chapters/01`、`/chapters/01/demos/direct`、`/chapters/01/demos/streaming`、`/chapters/01/demos/tool-call`。
- 本章右侧时序事件能展示对应的模型请求、模型响应和工具数据。
- 本章首条用户到应用事件能展示最小应用内部消息、三位数字会话 id 和字符串消息内容，并与后续模型 API 请求区分开。
- 本章 Responses API 工具结果回写请求采用无状态上下文重放方式。
- 本章三个演示都可以逐步推进，且未来消息线、未来传输和未来 payload 不提前出现。
- 本章流式输出演示能逐步累积助手回复，右侧时序图能用循环标记表示流式读取及处理。
- 本章左侧聊天状态和右侧时序图进度保持同步。
- 本章让应用的职责变得可见。
- 本章区分模型输出和应用工具执行。
- 本章不依赖后续概念也能被理解。
- 本章提供足够的交互方向，能支持后续架构工作。
