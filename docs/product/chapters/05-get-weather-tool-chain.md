# 05. 用 get_weather 拆解完整工具调用链路

## 复审边界

本文档是第 05 章课程内容合同。页面实现必须遵循全局章节编写要求：左侧只展示用户真实可见聊天，右侧时序图只展示真实发生的主链路事件，概念信息通过 payload、章节文字或独立演示说明呈现。

本章承接前序章节已经建立的推理接口、上下文和工具调用心智模型，只增加本章要讲的新机制。

## 学习目标

学习者完成本章后，应能解释：

- 参数缺失不应由模型凭空补全关键事实。
- 业务错误可以回写模型继续追问。
- 系统错误和异常数据应由工程层分类处理。
- 最终回答必须说明依据和不确定性。

## 原理讲法

工具调用链路需要同时设计成功路径和失败路径，否则模型会在不可靠数据上继续生成。

本章不把概念名词画成时序图参与者。时序图参与者只使用真实系统 actor、运行时、工具、外部系统或用户；本章机制通过请求 payload、工具结果、响应结构和章节说明展开。

## 具体实例

上海张江客户现场出行建议中，正常返回降水概率；城市缺失时追问；工具不可用时说明限制；降水概率 180% 时工程层拦截。

## 教学流程

- `/chapters/05/demos/weather-branches`：同一个天气工具的四种路径。正常、缺参、工具报错和异常数据在时序图上分开展示。
  1. 用户问题缺地点：用户没有说明城市，关键参数缺失。
  2. 模型追问地点：模型不应凭空填城市，而是追问客户现场在哪个城市。
  3. 正常工具返回：补齐地点后工具返回温度、降水概率、风力和时间戳。
  4. 工具暂不可用：数据源暂不可用属于可向用户说明的业务失败。
  5. 异常数据拦截：降水概率 180% 不应直接交给模型当事实使用。

## 交互与演示想法

- 左侧聊天只展示用户真实输入、助手思考状态和最终助手回答。
- 右侧标准时序图逐步展示用户、应用服务器、大模型、真实工具或外部系统之间的主链路事件。
- 当前步骤的关键上下文、参数、工具声明、工具结果、来源、审批状态或评估结果通过 payload 浮层展示。
- 反事实、错误示例或概念对照不插入主链路；需要展示时使用独立 demo 或 payload 注释说明。

## 边界与常见误解

- 本章不引入真实天气服务；全部使用 mock 工具返回。
- 不要把上下文、提示词、token 预算、参数、role、消息、缓存规则、技能说明或决策点画成时序图参与者。
- 不要把应用内部说明写进左侧聊天气泡。
- 不要引入真实外部 API、真实第三方账号、真实 OAuth、真实邮件或生产动作；课程页面使用 mock payload 展示机制。

## 内容复审验收标准

- 本章文档、课程目录和页面数据保持同一示例、同一主链路和同一核心机制。
- 本章左侧聊天只包含用户和助手真实可见消息。
- 本章主时序图只展示真实发生的系统事件。
- 本章 payload 能展示关键上下文或工程机制，而不是只展示标题。
- 本章不提前深入后续章节负责的机制。
