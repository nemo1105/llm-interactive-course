# 02. 多轮对话的真实机制

## 复审边界

本文档是第二章课程内容合同。它替换此前通用生成的第二章方案，要求第二章严格继承第一章普通对话演示的对象建模、时序图风格和 payload 质量。

本章只讲多轮对话最核心的一件事：模型不是天然记住网页里的聊天记录，而是应用服务器在下一次模型请求中把上一轮 `user` 和 `assistant` 消息显式带回去。

## 学习目标

学习者完成本章后，应能解释：

- 多轮对话仍然是一次又一次独立的模型调用。
- 第二轮回答能理解“那”，是因为第二次请求里包含第一轮用户消息和助手消息。
- 历史上下文不是时序图参与者，而是应用服务器构造模型请求时放进 payload 的 `messages` 或 `input`。
- 如果第二次请求只发送当前追问，模型看不到上一轮行程，只能追问或泛化回答。
- 显式历史重传会增加输入 token，并受上下文窗口限制。

## 原理讲法

本章在第一章普通对话基础上只增加一轮追问。时序图参与者保持为：

- 用户
- 应用服务器
- 大模型

历史消息只能在“应用服务器请求大模型”的 payload 中高亮展示，不能画成一个单独对象、服务或生命线。

## 具体实例

第一轮：

用户：“明天下午我要去上海客户现场，出门要带伞吗？”

助手：“建议带伞。2026-06-24 下午上海小雨转中雨，22-25℃，降水概率 82%……”

第二轮：

用户：“那我还需要提前多久出门？”

第二轮请求必须把第一轮 `user`、第一轮 `assistant` 和第二轮 `user` 一起发送给模型。模型基于这些可见 messages 才知道“那”指的是上一轮上海客户现场行程。

## 教学流程

1. 用户发送第一轮消息：左侧出现第一轮真实用户气泡；右侧显示用户消息进入应用服务器。
2. 应用服务器请求大模型：payload 展示第一轮 Chat Completions / Responses API 请求。
3. 大模型返回第一轮响应：payload 展示真实模型响应结构。
4. 应用服务器展示第一轮回答：左侧展示助手天气建议。
5. 用户发送第二轮追问：左侧新增“那我还需要提前多久出门？”。
6. 应用服务器带历史请求大模型：payload 展示第二次请求中的 `system`、第一轮 `user`、第一轮 `assistant`、第二轮 `user`。
7. 大模型返回第二轮响应：模型基于本次请求里的历史 messages 生成提前出门建议。
8. 应用服务器展示第二轮回答：左侧展示最终第二轮助手气泡。

## 交互与演示想法

- `/chapters/02/demos/history-replay`：多轮对话演示。
- 左侧只展示用户真实看到的聊天内容，不出现“应用”“历史上下文进入本次调用”等内部解释气泡。
- 右侧时序图只包含用户、应用服务器、大模型三个参与者。
- “历史”通过第二次请求 payload 中的 messages 高亮呈现。
- “如果不带历史会怎样”只作为章节文字中的原理说明，不进入主时序图消息、step 或左侧聊天状态。

## 边界与常见误解

- 不要把“历史上下文”画成参与者。
- 不要把应用服务器叫成“聊天应用”，保持第一章命名一致。
- 不要把内部教学说明放进左侧聊天气泡。
- 不要用“历史用户消息”“历史助手消息”这类占位字符串代替真实 messages。
- 不要在本章深入 RAG、memory 或历史压缩；这些概念只作为后续章节预告。

## 内容复审验收标准

- 本章时序图对象与第一章普通对话一致：用户、应用服务器、大模型。
- 本章是在第一章基础上增加第二轮追问，而不是另起一个抽象案例。
- 本章第二次模型请求展示真实 Chat Completions / Responses API payload。
- 本章左侧聊天只包含真实用户和助手消息。
- 本章主时序图只展示真实发生的两轮调用链路，不把反事实对照画成时序消息。
- 本章清楚说明模型理解“那”的原因是第二次请求看到了历史 messages。
